# ESP32C3-DEXP-AC-Protocol

Reverse-engineered UART protocol for a DEXP (Midea-like) air conditioner Wi-Fi dongle, plus a working ESP32-C3 bridge (`DEXP_AC_HomeKit_Bridge.ino`) that exposes the AC as an Apple HomeKit accessory using the [HomeSpan](https://github.com/HomeSpan/HomeSpan) library.

## Возможности моста (DEXP_AC_HomeKit_Bridge.ino)

- Эмулирует в Apple HomeKit аксессуар **HeaterCooler** (кондиционер/обогреватель)
- Управление: питание, целевая температура (16-32°C), режимы Heat/Cool, скорость вентилятора (Low/Medium/High), вертикальные жалюзи (Swing)
- Раз в 30 секунд отправляет пинг-пакет для получения актуального статуса от кондиционера
- Автоматически синхронизирует состояние в HomeKit при получении ответа от платы кондиционера (питание, режим, температура, скорость, свинг)
- ⚠️ Бридж использует не весь протокол — часть значений (Auto-скорость, Auto/Dry/Fan-Only режимы, горизонтальные жалюзи) поддерживается кондиционером, но пока не подключена в коде. Как их добавить — см. раздел «Конструктор» ниже.

## Подключение (Wiring)

ESP32-C3 подключается к плате управления кондиционера через UART (`Serial1`):

| Сигнал | GPIO ESP32-C3 |
|---|---|
| RX (приём) | 5 |
| TX (передача) | 6 |

Скорость: **9600 бод**, формат `SERIAL_8N1`.

## Установка

1. Установите библиотеку **HomeSpan** через Arduino Library Manager.
2. Выберите плату ESP32-C3 (Dev Module) в Arduino IDE.
3. Откройте `DEXP_AC_HomeKit_Bridge.ino` и укажите свои данные (см. ниже).
4. Прошейте плату, при первом запуске подключитесь к Wi-Fi точке HomeSpan (`HomeSpan-Setup`, см. документацию библиотеки) для первичной настройки, либо используйте `setWifiCredentials`.
5. Добавьте аксессуар в приложение **Дом** (Apple Home), введя код сопряжения.

## Настройка проекта (перед прошивкой)

В коде заполните плейсхолдеры своими значениями:

```cpp
homeSpan.setWifiCredentials("YOUR_WIFI_SSID", "YOUR_WIFI_PASSWORD");
...
homeSpan.begin(Category::AirConditioners, "AC1", "YOUR_HOMEKIT_PAIRING_CODE", "YOUR_SETUP_ID");
```

- `YOUR_WIFI_SSID` / `YOUR_WIFI_PASSWORD` — имя и пароль вашей Wi-Fi сети.
- `YOUR_HOMEKIT_PAIRING_CODE` — 8-значный код сопряжения HomeKit, который вы вводите в приложении Дом при добавлении аксессуара (формат `XXX-XX-XXX`, например `466-37-726`). Придумайте свой.
- `YOUR_SETUP_ID` — Setup ID из 4 символов (буквы/цифры), используется HomeKit для QR-кода/ручного ввода. Придумайте свой (например `AC01`).

⚠️ Не публикуйте реальные Wi-Fi данные и код сопряжения в открытом репозитории.

## Протокол UART (22-байтный пакет)

Пакет состоит ровно из **22 байт** (индексы 0–21). Компактная карта пакета — детали по отдельным полям вынесены в примечания под таблицей.

| # | Поле | Значения |
|---|---|---|
| 0 | Header | `0xAA` — всегда |
| 1 | Длина | `0x14` — всегда (20 байт данных после) |
| 2 | Тип пакета | `0x02` команда · `0x01` статус / пинг |
| 3 | Питание | `0x00` выкл · `0x01` вкл |
| 4 | Резерв ⁶ | `0x00` |
| 5 | Верт. жалюзи | `0x00` покачивание вкл · `0x06` выкл (зафиксировано) |
| 6 | Тип команды ¹ | `0x0B` опрос/пинг · `0x03` активный приказ |
| 7 | Режим ² | `0x00` Auto · `0x01` Cool · `0x02` Dry · `0x03` Heat · `0x04` Fan Only |
| 8 | Sleep/Eco | обычно `0x10`, либо `0x00` |
| 9 | Скорость вент. ³ | `0x00` Auto · `0x01` Low · `0x02` Medium · `0x03` High |
| 10 | Температура | `0x10`–`0x20` = 16–32 °C (прямое HEX-значение) |
| 11 | Гориз. жалюзи ⁴ | `0x1C` выкл · `0x1D`/`0x1E` вкл (разные положения) |
| 12–16 | Резерв ⁶ | `0x00` |
| 17–18 | Session ID ⁵ | см. примечание — не оставляйте статичным |
| 19–20 | Резерв ⁶ | `0x00` |
| 21 | Checksum | сумма байт 0–20 по маске `& 0xFF` |


**⁶ Резервные байты (4, 12–16, 19–20)** — во всех известных дампах всегда равны закодированному нулю. Протокол этого конкретного DEXP-донгла нигде публично не задокументирован (в отличие от более известных протоколов вроде Midea/Gree/Tuya, с которыми он не совпадает), поэтому назначение этих байтов остаётся неизвестным. Узнать его можно только эмпирически: сравнивая несколько реальных дампов с родного донгла, где эти байты отличаются между собой, либо перебирая их по одному на реальном оборудовании и наблюдая за реакцией кондиционера.

**¹ Байт 6** — родной донгл слал `0x03`, когда пользователь реально менял настройку (плата реагирует немедленно), и `0x0B` при обычном опросе статуса (кондиционер отвечает писком, но силовое реле может не щёлкать).

**² Байт 7** — стандартный HomeKit-сервис `HeaterCooler` поддерживает из коробки только Auto/Heat/Cool, поэтому Dry и Fan Only существуют в протоколе AC, но в текущем бридже не задействованы.

**³ Байт 9** — режим `Auto` (`0x00`) в протоколе есть, но бридж его пока никогда не отправляет и не распознаёт (см. «Конструктор» ниже).

**⁴ Байт 11** — второй, независимый свинг (влево-вправо). В бридже не управляется, значение статично.

**⁵ Байты 17–18** — «паспорт сессии». Родной донгл присылал разные пары: `12 31`, `12 32`, `17 2A`, `17 38` — похоже, менял их при каждом открытии приложения. ⚠️ Если слать туда фиксированный или «мёртвый» код, кондиционер через несколько недель может решить, что это залипшая кнопка, и перестать реагировать на команды.

Если сумма checksum не совпадает хотя бы на 1 — кондиционер полностью игнорирует пакет.

## Конструктор: что уже реализовано и как добавить недостающее

Бридж использует протокол не на 100% — часть значений из карты выше не подключена в текущей логике `update()` / `syncStates()`. Ниже — по каждому параметру: что реализовано сейчас и что поменять в коде, чтобы включить недостающий вариант.

### Скорость вентилятора (байт 9): нет Auto

Сейчас слайдер `RotationSpeed` имеет 4 положения (`setRange(0, 3, 1)`), но `0` используется как сигнал «выключить питание», а не как «Auto»:

```cpp
fanSpeed = new Characteristic::RotationSpeed(1);
fanSpeed->setRange(0, 3, 1);
...
if(fanSpeed->updated() && fVal == 0) { p = 0; active->setVal(0); }
byte fCode = 0x01;
if(fVal == 2) fCode = 0x02;
else if(fVal >= 3) fCode = 0x03;
else fCode = 0x01;
```

Чтобы добавить Auto как отдельное положение:

1. Уберите строку `if(fanSpeed->updated() && fVal == 0) {...}` — выключение оставьте только через основную кнопку питания (`active`).
2. Поменяйте маппинг скорости:
   ```cpp
   byte fCode;
   if(fVal == 0) fCode = 0x00;      // Auto
   else if(fVal == 2) fCode = 0x02; // Medium
   else if(fVal >= 3) fCode = 0x03; // High
   else fCode = 0x01;               // Low
   ```
3. В `syncStates()` расширьте обратный маппинг, чтобы Auto с кондиционера тоже отображался отдельно:
   ```cpp
   int fHome = (f == 0) ? 0 : (f == 2 ? 2 : (f == 3 ? 3 : 1));
   ```

### Режим работы (байт 7): нет Auto/Dry/Fan Only

Сейчас HomeKit-слайдер режимов ограничен только Heat/Cool:

```cpp
targState->setValidValues(2, 1, 2); // разрешены только 1 (Heat) и 2 (Cool)
...
byte mCode = (s == 1) ? 0x03 : 0x01; // Heat=0x03, Cool=0x01
```

Чтобы добавить Auto (у него есть родное значение `0` и в HomeKit, и в протоколе AC):

```cpp
targState->setValidValues(3, 0, 1, 2);
...
byte mCode = (s == 0) ? 0x00 : (s == 1 ? 0x03 : 0x01);
```

Dry (`0x02`) и Fan Only (`0x04`) не имеют аналога в стандартных значениях `TargetHeaterCoolerState` (там только Auto/Heat/Cool) — чтобы их добавить, нужна отдельная кастомная характеристика или сервис (например, доп. `Service::Fan` или `Switch`), это выходит за рамки текущей архитектуры на одном `HeaterCooler`.

### Тип команды (байт 6): всегда «опрос»

Сейчас пакет всегда шлёт `0x0B` (опросный код), даже когда это реальная команда пользователя:

```cpp
byte packet[22] = {
  0xAA, 0x14, 0x02, (byte)(p ? 0x01 : 0x00),
  0x00, swCode, 0x0B, mCode, 0x10, fCode, ...
```

Чтобы точнее повторить поведение родного донгла, замените `0x0B` на переменную и отправляйте `0x03`, когда пользователь реально что-то поменял:

```cpp
byte cmdType = (active->updated() || targState->updated() || fanSpeed->updated() || swing->updated()) ? 0x03 : 0x0B;
```

и используйте `cmdType` вместо `0x0B` в массиве `packet`.

### Горизонтальные жалюзи (байт 11): не управляются

Сейчас захардкожено `0x1D` (один из вариантов «включено»), переключателя в HomeKit нет:

```cpp
(byte)t, 0x1D, 0x00, 0x00, 0x00, 0x00, 0x00,
```

Стандартный сервис `HeaterCooler` не имеет второго свинга «из коробки» — под горизонтальные жалюзи нужен либо отдельный `Service::Switch` рядом с основным аксессуаром, либо оставить статичным значением (как сейчас). Если решите добавить переключатель — при выключенном состоянии подставляйте `0x1C` вместо `0x1D`.

### Session ID (байты 17–18): статичный код — риск блокировки

Сейчас зафиксировано:

```cpp
0x00, 0x05, 0x00, 0x00, 0x00
```

(байты 17 и 18 — это `0x00` и `0x05`)

⚠️ Так как в протоколе это «паспорт сессии», а родной донгл менял его при каждом запуске приложения, долгая работа со статичным значением может со временем привести к блокировке ID кондиционером. Практическое решение — задать случайную пару один раз при старте платы:

```cpp
byte sessionHi = 0x12, sessionLo = 0x31; // задайте один раз при старте, например случайно
```

и подставить `sessionHi, sessionLo` вместо `0x00, 0x05` в пакете.

### Уже реализовано без доработок

- **Питание** (байт 3) — полностью через `Active`.
- **Вертикальные жалюзи** (байт 5) — полностью через `SwingMode`.
- **Целевая температура** (байт 10) — весь диапазон 16–32°C.
- **Checksum** (байт 21) — считается автоматически, трогать не нужно.
