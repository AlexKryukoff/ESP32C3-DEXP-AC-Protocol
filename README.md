# ESP32C3-DEXP-AC-Protocol

Reverse-engineered UART protocol for a DEXP (Midea-like) air conditioner Wi-Fi dongle, plus a working ESP32-C3 bridge (`DEXP_AC_HomeKit_Bridge.ino`) that exposes the AC as an Apple HomeKit accessory using the [HomeSpan](https://github.com/HomeSpan/HomeSpan) library.

## Возможности моста (DEXP_AC_HomeKit_Bridge.ino)

- Эмулирует в Apple HomeKit аксессуар **HeaterCooler** (кондиционер/обогреватель)
- Управление: питание, целевая температура (16-32°C), режимы Auto/Heat/Cool, скорость вентилятора (0-3), жалюзи (Swing)
- Раз в 30 секунд отправляет пинг-пакет для получения актуального статуса от кондиционера
- Автоматически синхронизирует состояние в HomeKit при получении ответа от платы кондиционера (питание, режим, температура, скорость, свинг)

## Подключение (Wiring)

ESP32-C3 подключается к плате управления кондиционера через UART (`Serial1`):

| Сигнал | GPIO ESP32-C3 |
|---|---|
| RX (приём) | 5 |
| TX (передача) | 6 |

Скорость: **9600 бод**, формат `SERIAL_8N1`.

## Установка

1. Установите библиотеку **HomeSpan** через Arduino Library Manager.
2. Выберите плату ESP32-C3 (Dev Module) в Arduino IDE.
3. Откройте `DEXP_AC_HomeKit_Bridge.ino` и укажите свои данные (см. ниже).
4. Прошейте плату, при первом запуске подключитесь к Wi-Fi точке HomeSpan (`HomeSpan-Setup`, см. документацию библиотеки) для первичной настройки, либо используйте `setWifiCredentials`.
5. Добавьте аксессуар в приложение **Дом** (Apple Home), введя код сопряжения.

## Настройка проекта (перед прошивкой)

В коде заполните плейсхолдеры своими значениями:

```cpp
homeSpan.setWifiCredentials("YOUR_WIFI_SSID", "YOUR_WIFI_PASSWORD");
...
homeSpan.begin(Category::AirConditioners, "AC1", "YOUR_HOMEKIT_PAIRING_CODE", "YOUR_SETUP_ID");
```

- `YOUR_WIFI_SSID` / `YOUR_WIFI_PASSWORD` — имя и пароль вашей Wi-Fi сети.
- `YOUR_HOMEKIT_PAIRING_CODE` — 8-значный код сопряжения HomeKit, который вы вводите в приложении Дом при добавлении аксессуара (формат `XXX-XX-XXX`, например `466-37-726`). Придумайте свой.
- `YOUR_SETUP_ID` — Setup ID из 4 символов (буквы/цифры), используется HomeKit для QR-кода/ручного ввода. Придумайте свой (например `AC01`).

⚠️ Не публикуйте реальные Wi-Fi данные и код сопряжения в открытом репозитории.

## Протокол UART (22-байтный пакет)

Пакет состоит ровно из **22 байт** (индексы 0–21).

| Байт | Значение / диапазон | Описание |
|---|---|---|
| 0 | `0xAA` | Маркер начала пакета (Header), всегда статичен |
| 1 | `0x14` | Длина полезной нагрузки (HEX 14 = 20 байт данных после него) |
| 2 | Тип сообщения | `0x02` — управляющая команда (при изменении настроек); `0x01` — запрос статуса (пинг) / ответ от кондиционера |
| 3 | Питание | `0x01` — включен (ON); `0x00` — выключен (OFF) |
| 4 | `0x00` | Резерв / таймер, всегда нули |
| 5 | Вертикальные жалюзи (Vertical Swing) | `0x00` — включить покачивание (вверх-вниз); `0x06` — выключить (зафиксировать) |
| 6 | Сервисный байт / тип команды | `0x0B` — режим опроса / пинг / подтверждение статуса (пищит, реле может не щёлкать); `0x03` — активный приказ (заставляет плату немедленно реагировать) |
| 7 | Режим работы (MODE) | `0x00` Auto; `0x01` Cool; `0x02` Dry; `0x03` Heat; `0x04` Fan Only |
| 8 | Режим сна / экономичный | Обычно зафиксирован на `0x10` (или `0x00`) |
| 9 | Скорость вентилятора (FAN SPEED) | `0x00` Auto; `0x01` Low; `0x02` Medium; `0x03` High |
| 10 | Целевая температура | Прямое HEX-значение, диапазон 16–32°C. Примеры: `0x10`=16°C, `0x13`=19°C, `0x18`=24°C, `0x19`=25°C, `0x20`=32°C |
| 11 | Горизонтальные жалюзи и доп. флаги | `0x1C` — горизонтальный свинг выключен (базовое статичное состояние); `0x1D` / `0x1E` — горизонтальный свинг включен (разные траектории/положения) |
| 12–16 | `0x00` | Зарезервировано, всегда нули |
| 17–18 | Паспорт устройства / Session ID донгла | Критически важная зона. Родной донгл присылал маркеры сессии: `12 31`, `12 32`, `17 2A`, `17 38` — менялись при каждом открытии приложения. ⚠️ Если слать туда «мёртвые» нули или случайный статичный код (например `00 05`), кондиционер через несколько недель может заблокировать этот ID, посчитав его застрявшей кнопкой, и перестанет реагировать |
| 19–20 | `0x00` | Конец данных, всегда нули |
| 21 | Контрольная сумма (Checksum) | Сумма всех байтов 0–20 по маске `& 0xFF` (младший байт). Если сумма не совпадает хотя бы на 1 — кондиционер полностью игнорирует пакет |

### Реализация в `DEXP_AC_HomeKit_Bridge.ino`

Текущая реализация моста собирает пакет так:

```cpp
byte packet[22] = {
  0xAA, 0x14, 0x02, (byte)(p ? 0x01 : 0x00),
  0x00, swCode, 0x0B, mCode, 0x10, fCode,
  (byte)t, 0x1D, 0x00, 0x00, 0x00, 0x00, 0x00,
  0x00, 0x05, 0x00, 0x00, 0x00
};
```

Обратите внимание: байт 6 в текущей реализации всегда `0x0B` (опросный режим), а байты 17–18 зафиксированы как `00 05` — согласно протоколу выше это может со временем привести к блокировке ID кондиционером. Если начнутся проблемы с реакцией AC, стоит попробовать переключать байт 6 на `0x03` при реальных командах и/или ротировать session ID в байтах 17–18.

Пинг-пакет (запрос статуса, отправляется раз в 30 сек):

```cpp
byte ping[] = {0xAA, 0x02, 0x01, 0xAD};
```
